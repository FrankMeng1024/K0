# BUG-00003-STORY-00031: no-DB 模式 generate 端点无 GLM fallback，key 无效时阻塞主流程

**Priority**: Blocker
**Sprint**: Sprint 3
**Opened by**: QA (Stage 1 smoke test)
**Status**: Verified

## Description
在 no-DB 开发模式下（AUTH_ENABLED=false, DB 未配），主流程 Home → Learn → 粘贴文本 → GoalSelect → 快速了解 → Episode 屏 走到最后一步时，`GET /api/jobs/:id` 立即返回 `{status:'failed', error:'GLM_API_ERROR'}`，用户看到红色错误框和"重试"按钮，无法进入学习包页面。

## Steps to Reproduce
1. `BACKGROUND=1 bash scripts/start-backend.sh`（GLM_API_KEY=placeholder-set-real-key-for-ai-features）
2. `BACKGROUND=1 bash scripts/start-web.sh`
3. 浏览器打开 http://localhost:8081
4. 点击 Learn → 粘贴任意文本 → 开始 → 下一步 → 选择"快速了解"
5. 等 5-8 秒

## Expected
用户看到 6 步学习路径 + 卡片（buildMockPack 已经能造出完整 pack）。

## Actual
Episode 屏显示 "GLM_API_ERROR" 和"重试"按钮。

## Root Cause
`backend/src/routes/generate.js` 在 no-DB 模式下无条件调用 `generatePack()` → GLM API。当 `GLM_API_KEY` 为空、placeholder、或 GLM 服务返回 error 时，job 状态置为 `failed`，用户被阻塞。

对比 `snapshot` 端点（routes/snapshot.js）：no-DB 模式下 GLM 失败会返回 mock snapshot（见 BUG-00002 修复），generate 端点缺此 fallback。

## Fix
`backend/src/routes/generate.js` no-DB 分支（第 91-139 行）：
- 检测 `GLM_API_KEY` 为空或以 `placeholder` 开头 → 直接用 `buildMockPack(packId)` 造 pack，不调 GLM
- GLM 调用抛错时（catch 块）→ 用 `buildMockPack(packId)` 造 pack + 打 warning log，job 置为 ready 而非 failed
- 保留原真实 GLM 逻辑：GLM_API_KEY 有效时优先真实调用；仅在失败时回退

## Verification (post-fix)
1. 重启 backend
2. 重跑 happy path smoke test
3. 期待：8 秒内 Episode 屏出现 pack 数据（6 步 + 卡片）
