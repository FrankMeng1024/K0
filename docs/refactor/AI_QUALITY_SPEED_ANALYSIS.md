# K0 AI 速度 + 质量 深度分析报告

**日期**: 2026-07-10
**样本**: 2 次真实上传 (同一集 19 分钟职场播客 BYM #50, 峰哥&简里里, ~6000 字转录)
**方法**: 生产 DB (jobs + ai_call_logs) 拉真实耗时/token; 2 个独立 Opus subagent 根据原文对产出打分取平均
**质量总分**: **5.8 / 10** (评审A 6.05 + 评审B 5.5, 及格偏下)

---

## 一、速度分析 (真实生产数据)

| 阶段 | 耗时 | GLM 输入 tok | GLM 输出 tok | 其中 reasoning(思考) | 实际 JSON | reasoning 占比 |
|---|---|---|---|---|---|---|
| 快照 import | 116s (GLM 占 **83s**) | 11402 | 4130 | **2899** | 1231 | **70%** |
| 速学 quick | 32s (GLM 31s) | 2126 | 919 | 394 | 525 | 43% |
| 精学 deep | 168s (GLM 88s **首次失败** + 80s retry) | 2092 | 3602 | **2406** | 1196 | **67%** |

### 速度头号根因 (实锤): GLM-5.2 `reasoning_effort=max` 默认档
- **43%-70% 的输出 token 花在内部"思考"**上, 才吐 JSON。快照 4130 token 里 2899 是思考。
- 从快照的 reasoning trace 看到: GLM 把 oneSentence 改写了 4 遍、worthListening 段落逐字核对了 3 遍 —— 对"结构化抽取"任务, 这些思考大多是浪费。
- **提速杠杆 #1**: `enable_thinking=false` 或降 `reasoning_effort` → 预计砍掉 40-60% 延迟。快照 83s → 估 ~35s, 速学 31s → ~12s。

### 速度第二根因: 精学 retry 纯浪费
- 精学首次调用 88s 产出坏 JSON → retry 80s → 一共 168s, **一半时间浪费在失败的首次调用**。
- v49 已加 salvageTruncatedJson + 卡片上限 12, 这次 retry 成功了 (没再 GLM_MALFORMED_JSON), 但**首次仍坏**说明 prompt 还在诱导超长输出。
- **提速杠杆 #2**: prompt 进一步收紧 + 关思考档 (思考少 → 输出更快闭合 → 首次就成功, 省掉整个 retry)。

### 非瓶颈 (澄清)
- 转录 (BCUT ASR): 19 分钟音频轮询仅 ~10 次 <1s。**短播客转录不慢**。
- 长播客才会撞 ASR 轮询上限 (见第三节)。

## 二、质量分析 (2 subagent 独立打分, 严格对原文)

| 维度 | 评审A | 评审B | 均分 |
|---|---|---|---|
| 1 快照准确性 | 8 | 6.5 | 7.25 |
| 2 值得听/可跳过 | 8 | 6 | 7.0 |
| 3 卡片质量 | 6 | 5.25 | 5.6 |
| 4 概念解释器 | 5 | 6 | 5.5 |
| 5 行动计划 | 7 | 7 | 7.0 |
| **6 精学 vs 速学深度差** | **3** | **3.5** | **3.25** ← 最严重 |
| **总分** | **6.05** | **5.5** | **5.8** |

### 质量问题 (两评审一致确认, 按严重度)

**P0 — 精学卡片和速学【完全一样】(均分 3.25, 直接回答 Frank "精学没比速学深")**
- 逐条对比: quick 和 deep 的 4 张卡, **quote 一字不差、观点相同、insight 只换同义说法、deep 的 context 甚至更短**。
- 精学的"深"全靠**新增模块** (steps 6步 + concepts 概念 + actions 行动), 卡片本身**零增益甚至负增益**。
- 用户在"卡片视图"横向对比 → 看到两组一模一样的卡 → 判定"精学没更新、没更深"。**这是真缺陷不是错觉。**
- 根因: 两档卡片共用同一批 corePoints + 几乎同一 prompt, deep 没对卡片施加"更深"的独立要求。

**P1 — 概念"一句话解释"脱钩本集 (回答 Frank "习得性无助--一句话解释 很奇怪")**
- plain 文字本身没错 (标准词典定义), 但**是通用百科式解释, 没回扣本集嘉宾的因果链** ("提想法没 work 就受罚 → 给自己设了心理牢笼")。读起来像外接词条, 与播客脱节 = "奇怪"。
- related 字段空转 (同义反复); "学生思维"定义被"怕冲突"话题带偏窄化。

**P2 — quote 名为"金句"实为 AI 改写 (诚信级)**
- 所有卡片 quote 都是 AI 归纳的书面转述, **不是原文原话**。原文真金句 ("我习得性的无助了""像吃药一样去做""脑袋里有张地图") 反而没用。用户以为在读嘉宾原话, 实为二手概括。

**P3 — 一手叙事案例被系统性丢弃**
- 简里里"妈妈逼带 KPI 上大学/吃药式参加社团/30岁才开始运动"、峰哥"90%传统岗位不需要主动者、主动反受罚"、"整栋楼投票的食堂最丑"、ZOE"好设计师常和产品吵架" —— 这些**最生动、最可迁移、最反直觉**的内容全缺席。AI 偏爱抽象金句, 系统性忽略叙事/反直觉/限定条件型内容。

**P4 — 主题被"冲突"单点绑架 + context 同质化注水**
- 快照 4 条里 2 条讲冲突; 行动计划 3 档全讲冲突。原文另一半主线 ("相信一分耕耘一分收获"的心理建设、"逼自己做不喜欢的事"的自律) 被弱化。
- 4 张卡 context 句式雷同 ("嘉宾指出...本质上...真正的..."), 信息增量低。

### 亮点 (保持)
- 行动计划 today/week (7分): 具体可执行、有梯度。
- skippable 判断准 (片头/广告/片尾)。
- steps 里"批判性思考"那条有思辨性 (点出冲突需建立在共同目标+信任上)。

## 三、长播客问题 (50 / 90 / 150 / 220 分钟, Frank 补充)

当前样本仅 19 分钟。真实常见时长 50/90/150/220+ 分钟, 现设计**扛不住**, 三个层面:

### 3.1 转录 (ASR) — 会硬失败
- `bcut.js ASR_POLL_MAX=1800` = **30 分钟轮询天花板**。220 分钟音频 ASR 本身可能跑 >30 分钟 → **误判超时失败**。这是长播客**头号 Blocker**。
- 下载超时 `DL_TIMEOUT_LARGE=15min` + `fs.readFileSync` 整文件进内存 (220min≈200MB) → 慢 CDN 超时 / OOM 风险。

### 3.2 快照 GLM — 上下文够但会更慢
- GLM-5.2 是 **100 万 token 上下文** (已 context7 核实), 220 分钟转录 ≈ 5-10 万 token, 塞得下无溢出。
- 但输入越长, 快照 GLM 越慢 (19min 已 83s; 220min 输入翻 10 倍 + max 思考档 → 可能数分钟)。**关思考档对长播客收益更大**。

### 3.3 质量 — 长播客更稀释
- 现在 19 分钟就只抽 4 个 corePoints / 4 张卡, 覆盖不全 (漏了简里里整段故事)。
- 220 分钟播客信息量是 19 分钟的 10 倍, 若仍只出 4-12 张卡, **深度稀释到几乎没用**。
- prompt 的"动态密度"上限 12 张 (v49 为防 JSON 截断压低的) 对长播客**远远不够**。

### 长播客方向 (待实现, 本报告不写码)
1. **ASR 轮询/下载超时按音频时长动态化** (Blocker)
2. **流式读文件** 替代 fs.readFileSync (防 OOM)
3. **长转录分章 map-reduce**: 长播客先按章节 (15min/章) 并行抽要点, 再归并成快照 → 覆盖全 + 可并行提速 + 每次 GLM 调用输入可控
4. **卡片密度随时长放开** (关思考档 + salvage 兜底后, 不必再压 12 张)

## 四、改进优先级 (给 Frank 决策, 不自动执行)

| 优先级 | 改进 | 属性 | 预期收益 |
|---|---|---|---|
| **P0 速度** | GLM 关/降 `reasoning_effort` 思考档 | 参数 | 延迟 -40~60%, 快照 83s→~35s |
| **P0 质量** | 精学卡片走独立更深 prompt (同观点: 速学"是什么" / 精学"为什么+反例+边界"), 数量精学>速学 | prompt | 直接修 "精学没更深" |
| **P1 质量** | 概念 plain 强制回扣本集因果链 + 加生活化画面; related 给真实关系 | prompt | 修"解释很奇怪" |
| **P1 质量** | quote 强制摘原文原话 (禁改写); AI 归纳放 insight | prompt | 金句真实 |
| **P1 质量** | 抽取强制保留至少 1 段一手叙事 + 反直觉/限定条件观点 | prompt | 不丢血肉 |
| **P1 长播客** | ASR 轮询/下载超时动态化 + 流式读文件 | 配置/编排 | 220min 能跑通 |
| **P2 长播客** | 长转录分章 map-reduce + 卡片密度放开 | 编排+prompt | 长播客覆盖全+提速 |
| **P2 质量** | 主题覆盖均衡 (禁单一子话题挤占) + context 禁套话 | prompt | 去注水 |

**建议**: 开一个 "AI 提速+质量+长播客" Sprint, P0 两条 (关思考档 + 精学卡片独立 prompt) 先做, 收益最大且低风险 (纯参数/prompt, 不碰业务逻辑, 契合 AI 已独立成模块的架构)。本报告零代码改动。

## 附: 关键量化结论一句话
- **速度**: 慢在 GLM 思考档 (43-70% token 是思考) + 精学 retry 浪费, 不是转录。
- **质量 5.8/10**: 快照层能打 (7+), 但**精学卡片和速学一模一样 (3.25 分)** 是最大硬伤 —— 用户"精学没更深"的感受精准命中真缺陷。
- **长播客**: 220 分钟当前会在 ASR 轮询 30 分钟上限**硬失败**; GLM 上下文够 (1M) 但更慢 + 内容更稀释。
