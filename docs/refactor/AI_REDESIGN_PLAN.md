# K0 AI 重构方案 (速度 + 质量 + 长播客)

**日期**: 2026-07-10
**性质**: 头脑风暴后的完整设计方案, 待 Frank 逐条确认后再实施。零代码改动。
**依据**: 2 篇真实上传的速度/质量实测 (质量 5.8/10) + 3 个独立 subagent 联网调研 (结论高度一致)
**Frank 已定**: 精学=认知层跃迁+主动回忆; 长播客=一集一包多章(待最终确认); 先出完整方案再定顺序

---

## 一、核心诊断 (一句话)

K0 当前 AI 的三个痛点 —— 慢、精学不比速学深、长播客扛不住 —— **是三个错配, 不是模型不行**:
1. **档位错配**: 把 GLM-5.2 的"深度思考档"(默认 reasoning_effort=max)用在"快速抽取"任务上, 70% token 花在思考 → 慢。
2. **深度错配**: 把"精学"定义成"速学 + 更多模块"(数量差), 而非"高一个认知层次"(层次差) → 用户觉得没更深。
3. **范式错配**: 把 226 分钟长播客当一段文本一次性抽取 → ASR 超时 + 内容稀释。

---

## 二、模型选型: 不换, 换用法

**结论 (三方一致)**: 保留 GLM-5.2 (中文一流 / 1M 上下文 / 长程推理接近 Claude Opus / 开源可控成本)。当前问题在**档位**不在模型。

| 任务 | 档位 | 依据 |
|---|---|---|
| 快照 / 速学 / 抽取 / 主题分段 | `thinking=disabled` + `response_format=json` + `stream=true` | 抽取是归纳不是推理, 思考纯浪费。快照预计 83s→20-30s, 首字 2s 出现 |
| 精学核心生成 | 思考 `medium` (不用 max) + 流式 | 需要推理但不用顶配思考 |
| 未来脑图跨篇整合 | GLM-5.2 为主; 若洞察深度不足, 单点 (跨篇主题聚类/矛盾对比) 可补 Gemini(隐式缓存降本) 或 Claude Opus | 跨篇=长上下文+一致性+缓存复用 |

**关键收益**: 关思考档是**一个参数级改动、零迁移成本**, 直接砍掉 70% 浪费 + 让速学秒出。这是投入产出比最高的一步。

**验证方式**: 关思考档后, 用同一篇播客对比开/关的 (a)耗时 (b)质量分, 确认质量不掉再全量。

## 三、速学 vs 精学: 从"数量差"改成"认知层次差"

### 3.1 本质定位 (布鲁姆认知层次)

| | 速学 (~3min) | 精学 (~15min) |
|---|---|---|
| 认知层 | 记忆 + 理解 | 应用 + 分析 + 评价 |
| 阅读类比 | 高质量泛读 / 预习 | 精读 + 主动加工 |
| 回答的问题 | **这集讲了什么?** | **为什么成立? 哪里站不住? 我能怎么用? 我真懂了吗?** |
| 用户角色 | 被动接收(但精炼) | **主动思考 + 输出** |
| 成功标准 | 读完能一句话转述 | 能回答问题 + 能用到自己场景 |
| 留存率目标 | 10-20% | 50-90% |

**一句话口径**: 速学答"是什么", 精学答"为什么+怎么用+我懂了吗"。**深度来自"让用户动脑", 不来自"给用户更多"。**

### 3.2 速学重新设计 (目标: 3min 掌握全貌 + 能转述)

产出结构 (砍掉行动清单/概念详解, 那是精学的活):
1. **一句话核心** —— 费曼式大白话, 全集浓缩成 1 句 (决定用户能否转述)
2. **3 个关键要点** —— 严格 ≤3 个 (工作记忆短时接触取下限 3)
3. **1 个钩子** —— 最反常识/最有用的一点, 制造"想深入"的动机 → 天然引流精学
4. (保留) 值得听/可跳过时间轴 —— 帮判断"值不值得深入"

### 3.3 精学重新设计 (目标: 真内化, 主动加工) —— 本次重点

**关键: 精学的卡片必须是速学卡片的"下一层", 不是第二次归纳。** 按认知层次递进:

1. **带着问题进入** (费曼第一步): 先给 2-3 个"读完你应该能回答"的问题 —— 现在完全没有, 却是精读 vs 泛读最大的分水岭
2. **论证链拆解** (分析层): 不只给结论, 给"作者凭什么得出这个结论"(前提→推理→结论) —— 速学不做
3. **主动回忆点** (Active Recall): 关键处不直接给答案, **先抛问题让用户脑内回答, 再展开** —— 留存率 10%→50%+ 的最强杠杆, 几乎零成本
4. **批判视角** (评价层): 反驳 / 边界条件 / 隐含假设 / 与主流观点的冲突 —— 速学永远不会有, 是 5.8→8+ 的关键
5. **迁移应用** (应用层): 这个观点在你的什么真实场景能用、怎么用 —— 具体情境, 不是泛泛"行动清单"
6. **一句话费曼复述** (收尾): 让用户/AI 用大白话重讲一遍, 检验是否真懂

**精学卡片 vs 速学卡片的具体差异**:
- 同一个观点 (如"习得性无助"), 速学卡: quote(原话) + 一句话 insight + 2句 context (是什么)。
- 精学卡: 同观点 + **为什么成立(论证) + 一个反例/边界 + 与本集其他观点的关系 + 一个主动回忆问题** (为什么/怎么用)。
- **精学卡数量应 > 速学** (速学 3-5 张精选, 精学覆盖更全)。

### 3.4 顺带修的具体质量问题 (评审 P1-P4)

- **quote 用原文原话** (禁 AI 改写成书面语), AI 归纳放 insight 字段。修"金句失真"。
- **概念 plain 强制回扣本集** (第一句通用定义, 第二句必须引本集嘉宾的因果链) + 加一个生活化画面类比。修"习得性无助解释很奇怪"。
- **抽取强制保留 ≥1 段一手叙事** + 反直觉/限定条件观点。修"丢了简里里妈妈逼上大学那段最生动的内容"。
- **主题覆盖均衡** (禁单一子话题挤占快照/行动全部名额) + context 禁套话句式。

## 四、长播客切片 (50/90/150/226min): 分层 pipeline

### 4.1 产品形态: 一集一包多章 (Frank 待确认)

```
学习包 (一集播客)
├── 全局层: 一句话核心 + 全集脑图(章节间关系, = 未来脑图功能落点) + 3 要点
├── 章 1 (一个主题, 可独立速学/精学)
├── 章 2 ...
└── 收尾: 跨章整合(这几个主题连起来说明了什么 = 分析/评价层)
```
- 速学(长播客) = 只看全局层 (3min 掌握全貌)
- 精学(长播客) = 逐章攻 (一次 1-2 章, 每章正好落在 15min 健康微学习单元, 天然契合分次学习/间隔重复)
- 脑图 = "重新拼图"层, 把切散的章连回整体 (避免碎片化"豆腐块"陷阱)

### 4.2 技术分层 (ASR 按静音切, LLM 按主题切 —— 切点不同!)

```
226min 音频
├─【ASR 层】VAD 静音检测 → 切 5-15min 物理片 → 并行转录 → 时间戳对齐拼接
│    (解决 30min 轮询超时; 短片也走同管道, 只是少切)
├─【语义分段】便宜模型(关思考+JSON)对全文标主题切点 → N 个"章"(段标题+起止时间戳)
│    (逻辑切, 按内容; 和 ASR 物理切点不同)
├─【Map】每章独立生成 局部速学/精学/局部脑图节点 (并行, 关思考, 便宜)
└─【Reduce】长上下文模型 + 全局大纲 → 整合去重 → 全集脑图 + 分章学习包
     (跨多篇整合复用同一 Reduce, 输入换成多集脑图)
```

### 4.3 必做的工程根治 (长播客能否跑通的前提)

| 问题 | 现状 | 修 |
|---|---|---|
| ASR 轮询超时 | `ASR_POLL_MAX=1800` 写死 30min | 按音频时长动态 + VAD 切片并行 |
| 下载超时 | `DL_TIMEOUT_LARGE=15min` 写死 | 按时长动态 |
| **内存 OOM** | `fs.readFileSync` 整文件进内存(200MB) | **流式读 (createReadStream/ffmpeg 管道)** — 必做 |
| 短播客 | 单独逻辑 | 统一 pipeline, 按时长阈值分流(短的少切不走 map-reduce) |

### 4.4 ASR 方案: 锁定 BCUT + 切片 (路线 B 已否决)

**生产服务器实测: 4核 CPU / 3.6G 内存(仅 ~1.9G 可用) / 无 GPU。**
→ **路线 B (自建 FunASR) 否决**: 无 GPU 用 CPU 跑极慢, 且模型加载 1-2G 内存直接 OOM。撤回。

**锁定路线 A**: 继续用 BCUT (转录跑在 B站云端, 不占本机算力; Spike 已验证可转 4 小时长音频)。
- 226min 卡死的真根因是**我们代码 `ASR_POLL_MAX=1800` 写死 30min 上限**, 不是 BCUT 不行 → 按时长动态放大即可。
- 提速: ffmpeg 按静音切成几段, **并行提交 BCUT** (并行在 B站云端, 不占本机算力)。
- ⚠️ **流式读文件从"优化项"升级为"必做项"**: 本机仅 1.9G 可用内存, 220MB 音频用 fs.readFileSync 整个进内存会直接搞崩服务器。
- GLM 是云 API (智谱服务器), 关思考档只是改请求参数, 同样不占本机算力 → 速度/质量方案不受服务器限制。

## 4.5 处理顺序 (长篇 vs 短篇 — 已讲清)

**铁律**: AI 分章依赖转录文本 → **ASR 永远在 AI 之前**。不存在"先 AI 分段再 ASR"(没文字无法判断主题)。

**长篇 (≥ 时长阈值, 如 226min)**:
```
① 下载(流式读, 防 OOM)
② ASR 切片: ffmpeg 按【静音】切 ~15-20min 物理片 (纯为转录: 绕 30min 上限 + 并行)
③ 并行转录: 各片同时发 BCUT (并行在云端)
④ 按时间戳拼接 → 完整转录文本 (ASR 使命到此结束)
⑤ AI 分章: 便宜档读全文找【主题】切点 → N 章(标题+起止时间戳)  ← 按主题, 与②静音切点无关
⑥ AI 分析(Map): 每章独立生成 速学/精学 (并行, 关思考)
⑦ AI 归并(Reduce): 各章+全局大纲 → 全集脑图 + 跨章总结
   → 一集一包多章
```

**短篇 (< 阈值, 如 15-20min)**:
```
① 下载(同样流式读)
② 直接整段发 BCUT (15min ~10s, 不切)
③ 完整转录
④ 不分章(短+单主题) → AI 直接生成 速学/精学
   → 一集一包(单章, 即现形态)
```

**统一 pipeline**: 一个入口测时长 → 短的不切不分章单次生成 / 长的切片+分章+map-reduce。只在"要不要切片+分章"一个点分叉。阈值(25/30min)实测调。

**三个"切"别混**:
| | 切什么 | 按什么 | 目的 | 何时 |
|---|---|---|---|---|
| ASR 切片 | 音频 | 静音 | 转得动+快 | 转录前 |
| AI 分章 | 转录文本 | 主题 | 学习包分章 | 转录后 |

## 4.6 每阶段 模型 × 参数 (已实测/文档确认)

**统一**: 全程 GLM-5.2 (`glm-5.2`), 云 API, base `https://open.bigmodel.cn/api/coding/paas/v4`。不换模型只切参数。Air/Flash 暂不引入(coding 端点不一定有 + 当前非成本瓶颈, 需要时再加)。

**已实测确认的关键参数** (2026-07-10 用生产 key 实测):
- 关思考: `"thinking": {"type": "disabled"}` → 实测 reasoning_tokens 从数千归 **0**。
- 抽取任务关思考实测: 3观点+洞见抽取 **8.4s, 质量在线** (对比开思考档快照 83s, 快 5-10 倍不掉质量)。
- 强制 JSON: `"response_format": {"type": "json_object"}` (与 thinking 可叠加)。
- 流式: `"stream": true`。

| 阶段 | 任务 | thinking | response_format | stream | max_tokens | temperature |
|---|---|---|---|---|---|---|
| ① ASR | 转录(BCUT非GLM) | — | — | — | — | — |
| ② 快照 | 抽取 | **disabled** | json_object | true | 4096 | 0.3 |
| ③ AI分章(长篇) | 找主题切点 | **disabled** | json_object | false | 1024 | 0.2 |
| ④ 速学生成 | 抽卡(是什么) | **disabled** | json_object | **true** | 2048 | 0.4 |
| ⑤ 精学生成 | 认知层跃迁 | **enabled(开思考)** | json_object | true | 8192 | 0.5 |
| ⑥ Reduce归并(长篇) | 跨章整合+脑图 | enabled(开思考) | json_object | true | 8192 | 0.4 |

## 4.7 精学思考档: 已实测锁定 = 开思考

**实测结论 (2026-07-10, 同篇同 prompt, 仅切思考档, subagent 打分)**:
| 档 | 耗时 | 输出tok | reasoning | 质量分 |
|---|---|---|---|---|
| 开思考(enabled) | 67.5s | 4484 | 2737 | **8.5** |
| 关思考(disabled) | 31.8s | 1583 | 0 | 6.7 |

**精学锁定"开思考"** (差 1.8 分, 差在要害): 关思考会把精学降级成"更长的摘要"——批判性思考丢了"结构性vs心理建设"的归因思辨、漏了"心理建设"核心概念、quote 疑被润色失真。**慢一倍值得, 这是精学是否成立的分水岭。**
- 抽取类(②③④)= 关思考 (实测快 5-10 倍不掉质量)
- 精学⑤/归并⑥ = 开思考 (`thinking:{type:enabled}`)

## 4.9 用户思维原则 (贯穿实施 — Frank 强调)

- **更适合学习 + 提升效率**: 认知层跃迁(不堆量)、主动回忆(先问后答)、逐章可分次学。深度来自"让用户动脑"不是"给更多"。
- **页面不臃肿**: 精学信息量大, 但**不能一屏堆满**。设计上分层展开(全局→章→点)、按需展开、主动回忆问题折叠(点了才展开答案)。宁可用交互(展开/翻面/追问)承载深度, 不用"更长的文字墙"。
- **每加一个模块都问**: 这个对用户学习真有用吗? 还是花哨? 无用即删。

## 七、Frank 三个疑问的结论 (已讨论)

1. **短篇 ASR 切片提速?**: 收益很小 (BCUT 转 15min 仅 ~10s, 本就不慢), 不为提速切; 但统一 pipeline 保留切片能力 (短篇=切1片走同路)。真正提速大头在关 GLM 思考档。
2. **多章切片会不会乱/漏 (内容错位/跨主题总结)?**: 会, 是长播客最大技术风险。防法: (a)按语义切非按时长切 (b)允许一段内容重叠归属两章 (c)跨主题总结统一进"全局层/收尾"不塞单章 (d)Reduce 层兜底抓被切散的关联。诚实: 语义切片不可能 100% 完美, 设计留容错(宁冗余不遗漏), 需实测调。
3. **升级精学后速学卡保留?**: 不并存两套 (会造成"精学没更新"的困惑)。速学卡就地升级成更深的精学卡 + 补更多卡; 笔记必留, 收藏迁移。
