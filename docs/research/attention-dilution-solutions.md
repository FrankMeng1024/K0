# LLM 长文本注意力稀释：K0 播客场景解决方案调研

**调研日期**：2026-07-05
**目标模型**：GLM-5.2（1M context）
**问题**：45-99 分钟中文播客 transcript（24-67K tokens）单次处理时次要议题被稀释
**基线方案**：朴素 3 段分片（3x 耗时 + 3x 额度）

---

## 一、方案原理与成本对比

以「朴素 3 段分片」为 100% 基线。

### 1. 分片策略优化

| 方案 | 原理 | 耗时 | API 次数 | 质量 | K0 适用度 |
|---|---|---|---|---|---|
| **1a. 重叠切片（sliding window overlap）** | 段间保留 10-20% overlap 保连续性；后处理去重 | 105-120% | 3 次 | ↑ 上下文连续性，跨段议题不断裂 | 中：适合有跨段议题的播客（对谈类） |
| **1b. 语义边界切片** | 按话题自然段落切（先跑一次轻量话题检测），非固定时长 | 110-130% | 3-4 次 | ↑ 议题完整性，避免"腰斩" | 高：播客本身话题跳跃，适配好 |
| **1c. 两阶段递归** | Pass 1：全文一次调用只输出议题清单（低 max_tokens）；Pass 2：仅对具体议题精读原文 | **60-80%** | 1 + N 次（N=议题数，通常 3-6） | ↑↑ 议题不遗漏 + 每个议题深挖 | **最高**：K0 议题少而独立，天然适配 |

### 2. 单次调用 + Prompt 技巧（不分片）

| 方案 | 原理 | 耗时 | API 次数 | 质量 | K0 适用度 |
|---|---|---|---|---|---|
| **2a. 议题清单先行（"outline-first"）** | 强制 LLM 先输出议题目录再逐条填充；结构化 JSON schema | **30-35%** | 1 次 | ↑ 覆盖率，↓ 稀释；但仍受 U 型注意力影响 | **高**：成本最低，实测 K0 已在做 |
| **2b. 重复关键指令（bookending）** | "不要遗漏次要议题"放 system prompt 开头 + user prompt 结尾 | 30-35% | 1 次 | ↑ 边际，非治本 | 中：作为其他方案的增强项 |
| **2c. Chain-of-Density (CoD)** | 迭代生成：先粗（覆盖）→ 再压缩（增密），5 步迭代 | **150-200%** | 5 次 | ↑↑ 信息密度 | 低：为压缩摘要设计，不解决稀释；耗时反而更高 |
| **2d. 结构化 JSON schema 强约束** | 用 `topics: [{title, importance, timestamp, detail}]` 强制格式，`importance=minor` 必须至少 2 条 | 30-35% | 1 次 | ↑ 强制枚举次要议题 | **高**：与 2a 组合几乎无成本收益最好 |

### 3. 检索增强（RAG）

| 方案 | 原理 | 耗时 | API 次数 | 质量 | K0 适用度 |
|---|---|---|---|---|---|
| **3a. 议题驱动 RAG** | Pass 1：低成本扫全文出议题清单（≤ 2K tokens 输出）；Pass 2：向量化 transcript → 每议题 retrieve top-k 段落 → 精读 | **80-120%** | 1 + N + 向量化 | ↑↑ 精度高、可扩展至 3 小时+ | 中：需引入向量库（Chroma/Qdrant），工程复杂度上升 |
| **3b. BM25 + 议题检索**（无向量化） | Pass 1 出议题+关键词；Pass 2 用 BM25 抽段落再精读 | **60-90%** | 1 + N | ↑ 中文关键词命中率高 | **高**：零依赖、快、成本最低的 RAG 变体 |

### 4. 模型侧

| 方案 | 说明 | K0 适用度 |
|---|---|---|
| **4a. GLM-5.2 参数调优** | `temperature=0.3, top_p=0.85`：更聚焦不发散；`max_tokens` 分配给结构化字段 | **高**：零成本 |
| **4b. Claude 200K/Gemini 2M** | 长文本注意力显著优于 GLM（业界共识） | 不适用（合规） |
| **4c. 静态 prompt caching** | 智谱开放平台已支持 prompt 缓存（system prompt 部分复用） | **高**：批量处理时节省 30-50% 输入 token |

### 5. 中英混合特殊处理

- **现象**：GLM 对中文里嵌入的英文术语（"Skills"、"Agent"、"Prompt"）分词正常，但**大写单字符缩写**（"K"、"AI"）在长上下文易被视作背景噪声跳过
- **建议**：
  - Prompt 明确声明："transcript 中的英文术语（Skills、Agent、Prompt 等）为专有名词，必须原样保留在议题标题中"
  - 输出 schema 加 `keywords_en: [string]` 字段，强制模型抽取英文关键词
  - 预处理：ASR 转录后用简单规则把连续大写英文（≥ 2 字符）加 `『』` 标记，帮 LLM 视觉聚焦

## 二、推荐 Top 3 方案

### 最省成本：**方案 2a + 2d + 4a**（结构化 outline-first + JSON schema + 参数调优）
- **成本**：30-35% 基线（单次调用）
- **质量**：覆盖率显著提升，但仍受 U 型注意力影响，最长的 99 分钟播客次要议题仍可能漏 1-2 条
- **实施**：改 prompt + 加 JSON schema + 调 temperature，**当天可上线**
- **适合**：K0 早期，对稀释容忍度较高

### 平衡方案（**推荐**）：**方案 1c 两阶段递归**
- **成本**：60-80% 基线
- **质量**：与朴素分片相当或更好——Pass 1 全文视角保证不漏议题，Pass 2 局部精读保证深度
- **实施**：拆两次 API 调用，Pass 1 用 `max_tokens=1500` 只输出议题清单 JSON，Pass 2 循环调用 3-6 次
- **风险**：Pass 1 若漏议题则 Pass 2 无法补救 → 用 2d 结构化 schema 约束议题最小数量
- **适合**：K0 生产环境，性价比最优

### 最高质量：**方案 3b（BM25 议题驱动 RAG）+ 4c（prompt caching）**
- **成本**：60-90% 基线，批量处理时更低
- **质量**：可扩展至 3 小时以上超长播客，理论最优
- **实施**：引入 rank_bm25（Python 库，零依赖），Pass 1 出议题+关键词，Pass 2 检索段落精读
- **风险**：工程复杂度 +1（分词、索引），需 1-2 周实施
- **适合**：K0 中期，当需要处理 90+ 分钟播客成为常态时

## 三、中英混播客特殊 tips

1. **prompt 显式声明**："播客含中英混合表达，英文术语（Skills/Agent/Prompt/AI 等）为专有名词，请在议题标题中原样保留，不要翻译"
2. **输出 schema 加 `keywords_en`**：强制抽取，避免英文术语被视作噪声
3. **ASR 转录预处理**：连续大写英文用 `『』` 或 `**` 包裹，帮 LLM 视觉聚焦
4. **议题清单校验**：Pass 1 出议题后，简单规则校验——如果全文英文单词占比 >5% 但议题清单里 0 个英文关键词 = 漏了，触发重试
