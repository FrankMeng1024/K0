# 长文本注意力衰减：Prompt 缓解方案实测

**日期**：2026-07-05
**测试对象**：GLM-5.2（open.bigmodel.cn coding endpoint）
**测试语料**：跨国串门 99min 播客 transcript（2621 segments / 67577 chars）
**验证锚点**："加州选举诚信"议题——出现于原文 4176-5945s（后 30% 区间），最容易被单次调用漏掉的次要独立议题

## 背景

用户之前跑单次 5.2 处理这集时反馈**漏了"加州选举"议题**。朴素分片能救回来但 3× 耗时。本次实测三种"单次调用改 prompt"方案能否稳定捕捉。

## 实验设计

三种改进方案 + baseline，各方案独立运行 1-2 次以观察随机性：

- **Baseline**：K0 现有 prompt，单次调用，无任何加强
- **方案 A**：system prompt 强制先输出 `topicInventory`（10-15 条议题清单），再展开学习包主体
- **方案 B**：system prompt 开头+结尾双次强调"遍历所有议题"，transcript 里插入 `[章节 N: MM-MM min]` 标记（每 15 分钟 1 章节，共 7 章节）
- **方案 C**：两轮 API 调用——Round 1 只列议题清单，Round 2 把 clean + 全文 + Round 1 输出一起交给模型展开

## 关键发现

**1. Baseline 不是必然漏，是概率性漏**
3 次 baseline 里，2 次明确讲了"加州选举"（step 6 展开），1 次只讲了泛化"民主制度风险"没锚定加州。用户之前"漏了"是这条概率分布的尾部。想要**稳定**捕捉必须加干预。

**2. 方案 A 的陷阱：清单命中 ≠ 主体展开**
方案 A 强制先列 topicInventory，两次都完美列出加州选举（第 14-15 条）。但**用户实际读到的 steps/cards 6+3 主体展开里，两次都没有加州选举**——所有 step 时间戳集中在前 55 分钟。清单里"关键词命中"是假象，真正决定"用户学到什么"的是 steps/cards。而且 A 因为要多输出清单，out token 翻倍（2607 → 5062），耗时也翻倍（70s → 119s），性价比最差。

**3. 方案 B 是唯一同时做到"清单意识 + 主体展开 + 时间戳分布均匀"的方案**
两次 B 的 6 个 steps 覆盖 641s / 1046s / 2007s / 2720s / 3690s / 4238s，跨度 76 分钟，最后一步稳定落在加州选举。章节标记 `[章节 N: MM-MM min]` 起到了显式空间锚点作用，配合"6 步必须分布在不同章节"的硬约束——**结构化约束比重复说教更有效**。out token 甚至比 baseline 更省（2042 vs 2607）。

**4. 方案 C 稳定但昂贵**
两次 C 都命中且主体展开，但 Round 1 因 `max_tokens: 2000` 被截断（清单太长导致 JSON 破损）。有趣的是 **Round 1 输出为空 Round 2 依然能正确展开**——说明 Round 2 那次调用因上下文里"我刚被要求列清单"的隐含指令，自身就完成了清单扫描。这暗示 C 的效果 ≈ B + 更贵的实现。耗时 150-163s，是 baseline 的 2.3×，两次 API 调用累计 in tok ≈ 9 万（B 只有 4.5 万）。

**5. 与朴素分片对比**（参考 spike-016 已有数据）
朴素分片 3 段并发 ≈ 90-120s，成本 3× baseline。方案 B 76s / 单次调用 / 45k in tok / 2k out tok——**成本仅为分片 1/3，耗时更短，主体展开质量相当**。

## 完整命中率总结

| | 清单/关键词命中 | 主体展开命中 |
|---|:---:|:---:|
| Baseline | 2/3 | 2/3 |
| 方案 A | 2/2 | **0/2** |
| 方案 B | **2/2** | **2/2** |
| 方案 C | 2/2 | 2/2 |

**方案 B 是唯一一个"清单命中率" 与"主体展开命中率"都是 100% 的方案**，且成本最低。

## 推荐

**K0 生产用方案 B**（重复强调 + 章节标记）。理由：

1. 主体命中率 2/2（样本小但趋势明确），显著优于 baseline 2/3 和方案 A 的 0/2
2. 单次调用，耗时 60-80s（与 baseline 相当），远低于朴素分片
3. Token 成本几乎与 baseline 持平（in +200, out -500 到 +80）
4. 实现简单：只需改 system prompt 头尾 + 在拼接 transcript 时插入章节标记，无需两轮 API 调度
5. 副产物：6 步 sourceTimestamp 强制均匀分布，学习包整体质量提升

**次选**：如果 K0 未来遇到 90 分钟以上更长的 transcript 或有更严的漏项风险，可用方案 C 作为 fallback（更贵但更保险）。

**不推荐**：方案 A。清单命中给运营/日志层假的"覆盖幻觉"，用户实际读到的学习包主体仍会漏。且成本翻倍。

## 诚实标注

- **成功**：方案 B（清单+主体双命中，成本最低）；方案 C（命中但贵 2×）
- **失败**：方案 A——虽然议题清单完美但主体展开完全没解决问题，比 baseline 更糟（浪费 2× token 换来了假的关键词命中）
- **样本量限制**：每方案只跑 1-2 次，无法给出严格统计显著性。B 的 2/2 命中在小样本下是强信号但不是保证。生产上应加**运行后校验**：output 里 6 个 step 的 sourceTimestamp 覆盖至少 4 个不同 15min 章节，否则触发一次重试
- **验证锚点单一**：只测了"加州选举"这一个后段议题。理论上 B 应对其他"后段独立议题"也有效，但未跨语料验证。建议 K0 上线后灰度观察
